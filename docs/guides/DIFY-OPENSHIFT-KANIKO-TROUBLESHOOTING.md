# Dify Enterprise on OpenShift — Kaniko Plugin Builder Permission Fix

> **Status**: Active investigation (May 2026)
> **Cluster**: `pcs.sandbox2284.opentlc.com`
> **Namespace**: `pcs-dev`
> **Dify Edition**: Enterprise (uses `enterprise.dify.ai/v1` CRDs)

## Problem Summary

When installing plugins via the Dify Enterprise UI, the plugin_connector creates a
Kubernetes Job that uses **Kaniko** (`gcr.io/kaniko-project/executor:latest`) to build
a container image of the plugin. This Job fails on OpenShift with:

```
ERROR: Could not install packages due to an OSError: [Errno 13] Permission denied: '/app/.local/lib'
```

The error progressively moves deeper into the path as workarounds are applied:
- `/app/.local` → `/app/.local/lib` → `/app/.local/lib/python3.12`

## Root Cause Analysis

### What happens during plugin installation

1. User clicks "Install Plugin" in the Dify Enterprise UI
2. The **plugin_connector** deployment creates a `DifyPlugin` CR (`enterprise.dify.ai/v1`)
3. A Kubernetes **Job** is created with a Kaniko executor container
4. Kaniko downloads the build context from S3 (MinIO), executes the Dockerfile, and pushes the built image to Harbor
5. The plugin runs from the built image

### Why it fails on OpenShift

The failure chain has **two layers**:

**Layer 1 — Kaniko itself runs as root (solved by `anyuid` SCC)**

The pod YAML confirms the pod was admitted via `anyuid` SCC and runs as `uid: 0`:
```yaml
annotations:
  openshift.io/scc: anyuid
# ...
user:
  linux:
    uid: 0
    gid: 0
```

This part is working — Kaniko itself starts successfully.

**Layer 2 — The Dockerfile built by Kaniko has a `USER` directive (the actual problem)**

The Dify plugin Dockerfile (generated by plugin_connector) likely follows this pattern:
```dockerfile
FROM python:3.12-slim
WORKDIR /app
COPY plugin-code/ .
USER 1000           # <-- switches to non-root user
RUN pip install ...  # <-- FAILS: non-root user can't write to /app/.local/lib
```

Kaniko faithfully executes the Dockerfile instructions, including `USER` directives.
When Kaniko processes `RUN pip install` after `USER 1000`, it runs pip as UID 1000,
which does not have write permissions to `/app/.local/lib`.

This is NOT an OpenShift-specific bug — it would also fail on vanilla Kubernetes.
The confusion arises because people assume the issue is about the *pod's* security
context, when it's actually about the *Dockerfile's* `USER` directive during the
*build process*.

### Why Kah Hoe's custom Kaniko images didn't fully fix it

The debug images (`quay.io/kahlai/executor:debug`, `debug2`, `debug3`) appear to
pre-create directories in the Kaniko executor image itself. But Kaniko extracts the
**base image filesystem** (e.g., `python:3.12-slim`) and operates on *that* — not
its own image filesystem. Pre-creating `/app/.local/lib` in the Kaniko executor
image has no effect on the build context filesystem.

The progression of errors (`/app/.local` → `/app/.local/lib` → `/app/.local/lib/python3.12`)
suggests that each debug image created one more level of the directory tree, but the
underlying permission problem persists deeper.

## Solutions (Ordered by Recommendation)

### Solution 1: Modify Dockerfile via Helm Values (Recommended)

If Dify Enterprise Helm chart exposes the plugin Dockerfile template or allows
injecting build args, ensure `pip install` runs as root:

```dockerfile
# Correct ordering:
FROM python:3.12-slim
WORKDIR /app
COPY plugin-code/ .
RUN pip install --no-cache-dir -r requirements.txt   # runs as root (default)
RUN mkdir -p /app/.local && chown -R 1000:1000 /app/.local
USER 1000
```

Check the Helm values for the plugin_connector:
```bash
helm show values dify/dify | grep -A 20 plugin
```

Look for values like:
- `pluginConnector.builder.dockerfile` or similar template overrides
- `pluginConnector.builder.args` for extra Kaniko arguments

### Solution 2: Add `--build-arg` to Force Root During pip install

If the Dockerfile uses an `ARG` for the user, override it in the Kaniko args.
In the Helm values, add extra args to the Kaniko builder job:

```yaml
pluginConnector:
  builder:
    extraArgs:
      - "--build-arg=USER=root"
```

### Solution 3: Set SecurityContext on the Builder Job

Ensure the Job template has the correct security context AND a ServiceAccount
bound to the `anyuid` or `privileged` SCC:

```bash
# Create a dedicated ServiceAccount
oc create sa dify-builder -n pcs-dev

# Grant anyuid SCC (minimum required for Kaniko)
oc adm policy add-scc-to-user anyuid -z dify-builder -n pcs-dev

# If anyuid is not enough, try privileged (use with caution)
oc adm policy add-scc-to-user privileged -z dify-builder -n pcs-dev
```

Then configure the Helm chart to use this ServiceAccount for builder jobs:
```yaml
pluginConnector:
  builder:
    serviceAccountName: dify-builder
    podSecurityContext:
      runAsUser: 0
      fsGroup: 0
```

> **Note**: The current Job YAML shows `serviceAccountName: default` and
> `securityContext: {}`. The `anyuid` SCC is already being applied, but this
> only affects the Kaniko *process*, not the Dockerfile `USER` directives.

### Solution 4: Use Buildah Instead of Kaniko (If Dify Supports It)

If the Helm chart allows changing the builder image, Buildah is a better fit
for OpenShift:

```yaml
pluginConnector:
  builder:
    image: quay.io/buildah/stable:latest
```

Buildah advantages on OpenShift:
- Red Hat native tool, designed for OpenShift
- Supports rootless builds with user namespaces
- Doesn't require `USER` workarounds
- Compatible with the existing Dockerfile format

**Caveat**: Dify's plugin_connector likely generates Kaniko-specific CLI args
(`--dockerfile`, `--context`, `--destination`). Buildah uses different syntax, so
this would require changes in the plugin_connector code itself.

### Solution 5: Build a Custom Kaniko Image with Workaround (Quick Fix)

Build a custom Kaniko image that wraps the executor and ensures permissions are
correct in the build filesystem, not the executor filesystem:

```dockerfile
FROM gcr.io/kaniko-project/executor:latest

# Override the entrypoint to fix permissions before building
COPY <<'WRAPPER' /kaniko/wrapper.sh
#!/busybox/sh
# Kaniko extracts the base image to / — ensure /app/.local is writable
mkdir -p /app/.local/lib/python3.12
chmod -R 777 /app/.local
# Run the actual Kaniko executor
exec /kaniko/executor "$@"
WRAPPER

RUN chmod +x /kaniko/wrapper.sh
ENTRYPOINT ["/kaniko/wrapper.sh"]
```

Build and push this image:
```bash
podman build -t harbor.apps.pcs.sandbox2284.opentlc.com/dify/kaniko-openshift:latest .
podman push harbor.apps.pcs.sandbox2284.opentlc.com/dify/kaniko-openshift:latest
```

Then configure the Helm chart to use it:
```yaml
pluginConnector:
  builder:
    image: harbor.apps.pcs.sandbox2284.opentlc.com/dify/kaniko-openshift:latest
```

**Why this works**: Unlike the previous debug images that pre-created directories in
the Kaniko image's own layers, this wrapper script runs *after* Kaniko extracts the
base image filesystem but *before* executing Dockerfile instructions. The `mkdir` and
`chmod` operate on the actual build filesystem.

> **Important note on timing**: This wrapper approach has a race condition — Kaniko
> extracts the base image AFTER the entrypoint runs. A more robust approach is
> Solution 6.

### Solution 6: Patch the Kaniko Executor to Ignore USER Directives (Nuclear Option)

Kaniko has an undocumented/experimental flag in some forks:

```
--run-as-user=0
```

This forces all `RUN` instructions to execute as root regardless of `USER` directives
in the Dockerfile. Check if the community fork (`osscontainertools/kaniko`) supports
this:

```bash
# The original Google kaniko repo was archived in June 2025
# The maintained fork is at: https://github.com/osscontainertools/kaniko
# Image: martizih/kaniko
```

If supported, add it to the Kaniko args in the Helm chart:
```yaml
pluginConnector:
  builder:
    extraArgs:
      - "--run-as-user=0"
```

1. **Inspect the actual Dockerfile** being built by Kaniko:
   ```bash
   # Get the build context from S3 and extract the Dockerfile
   oc exec -n pcs-dev deploy/plugin-connector -- \
     cat /path/to/connector_build_caches/1e15a22cbbbbd2a4897118299de59857.tar | tar -xf - Dockerfile
   # Or download from MinIO directly:
   mc cp minio/dify/connector_build_caches/1e15a22cbbbbd2a4897118299de59857.tar /tmp/
   tar -xf /tmp/1e15a22cbbbbd2a4897118299de59857.tar Dockerfile
   cat Dockerfile
   ```
   This will confirm whether the Dockerfile has a `USER` directive before `pip install`.

2. **Check Dify Enterprise Helm values** for builder configuration:
   ```bash
   helm get values dify -n pcs-dev -a | grep -A 30 -i "plugin\|connector\|builder\|kaniko"
   ```

3. **Check if the Dockerfile can be customized** via the plugin_connector config:
   ```bash
   oc get configmap -n pcs-dev -l app=plugin-connector -o yaml
   ```

4. **Try Solution 5** (custom Kaniko wrapper image) as a quick validation,
   then move to Solution 1 or 3 for a production fix.
